## Why

Ito currently stores shared workflow state in markdown and git, which works for single-session flows but becomes unreliable when multiple harnesses work from different worktrees and stale snapshots. We need a project-scoped API so agents can coordinate ownership, read current artifacts, and sync updates without depending on git timing or branch state.

## What Changes

- Add a new OpenAPI-defined backend API for shared Ito state, scoped by project identity.
- Add token-based authentication for backend access (project-scoped bearer token via environment variable).
- Add change-level lease locking with heartbeat and explicit release to prevent duplicate work on the same change.
- Add server-side markdown artifact storage (proposal, design, tasks, and spec deltas) with optimistic concurrency.
- Add backend change lifecycle state so archived changes are clearly marked and become immutable.
- Add change allocation endpoints so an agent can atomically claim the next available change.
- Add event ingest endpoints for audit/state-transition events generated by harnesses.
- Add backend configuration keys in Ito config and environment resolution rules for URL, project ID, and token.

## Capabilities

### New Capabilities

- `backend-state-api`: Versioned OpenAPI contract for project-scoped Ito state management.
- `backend-auth`: Bearer-token authentication and authorization model for API access.
- `change-leasing`: Lease/heartbeat/release semantics for exclusive change ownership.
- `change-allocation`: Atomic selection and claim of next available unlocked change.
- `backend-artifact-store`: Markdown artifact persistence with revision-based conflict handling.
- `backend-event-ingest`: Endpoint contract for receiving structured audit/state events.

### Modified Capabilities

- `config`: Add backend configuration schema and environment variable mapping for connection and auth.

## Impact

- **Affected code**: New backend service crate(s), DTO/schema definitions, persistence adapters, config loading, and integration tests.
- **Affected APIs**: New versioned HTTP API (`/v1/...`) with OpenAPI document publishing.
- **Affected storage**: New backend datastore for project metadata, leases, artifacts, and event records.
- **Operational impact**: Introduces backend deployment/runtime concerns (availability, token management, data backups).
- **Workflow impact**: Enables multi-agent coordination through explicit change claims rather than implicit branch/worktree conventions.
